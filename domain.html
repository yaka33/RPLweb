Research Domain

Internet of Things (IoT) Network Security - RPL Routing Protocol Trust Management

This research sits at the intersection of embedded systems security, wireless sensor network protocols, and distributed trust management. Specifically, it addresses security vulnerabilities in IPv6 Routing Protocol for Low-Power and Lossy Networks (RPL), the IETF-standardized routing protocol for IEEE 802.15.4 IoT networks, operating on resource-constrained nodes with as little as 32KB RAM and 8MHz CPU.

Literature Survey

Existing research on RPL security has largely followed two directions. The first involves cryptographic approaches using digital signatures or message authentication codes to verify routing advertisements. While effective, these approaches impose prohibitive computational and memory overhead on constrained IoT devices. The second direction explores trust-based routing extensions to RPL, where neighbor trust scores influence parent selection.

Notable works include THC-RPL (Mushtaq et al., 2022), which introduced lightweight trust-enabled routing against Sybil attacks, and Zhou et al. (2025), who proposed a reliable trust objective function for RPL. Fuzzy logic approaches such as FSRW (Singh & Kaur, 2024) combined trust with optimization algorithms for healthcare IoT. In the intrusion detection space, Khan et al. (2023) surveyed computational intelligence techniques for WSN intrusion detection, while self-healing frameworks such as Antilizer (Mbarek et al., 2018) explored runtime security repair for wireless sensor networks.

However, a critical gap persists: no existing lightweight framework simultaneously addresses trust computation, attack detection, distributed isolation, and trust-aware routing enforcement within a single, energy-adaptive, fully distributed architecture targeting Contiki-NG RPL networks.

Research Gap

Current solutions for RPL security suffer from one or more of the following limitations:

  • Single Attack Focus - They address only one attack type (e.g., Sybil or Sinkhole) rather than simultaneous multi-vector attacks
  • Centralized Dependency - They require centralized components (base station or sink) that become single points of failure
  • Computational Intensity - They use ML-based detection that is computationally infeasible on nodes with 32KB RAM
  • Static Models - They apply static trust computations without considering real-time energy constraints or hardware health
  • Lack of Feedback Loop - Trust scores are computed but not fed back into the routing protocol's objective function
  • No Recovery Path - Once a node is penalised, there is no mechanism for it to return to a trusted status

No existing work integrates multi-metric trust computation, statistical IDS, distributed consensus-based isolation, and a custom RPL Objective Function into a single, energy-adaptive, self-healing framework.

Research Problem

"How can a fully distributed, resource-constrained IoT network running RPL autonomously detect simultaneous Sinkhole and Selective Forwarding attacks, quantify multi-dimensional node trustworthiness, collaboratively isolate malicious nodes without centralized authority, and enforce trust-aware routing decisions - all within the strict energy, memory, and CPU limits of IEEE 802.15.4 sensor nodes?"

Research Objectives

ID Objective Description
SO1 Design and implement an energy-adaptive trust computation engine that evaluates node trustworthiness across four dimensions (link, energy, behavior, thermal) and adjusts frequency based on battery status.
SO2 Develop a lightweight, distributed IDS capable of simultaneously detecting Sinkhole and Selective Forwarding attacks using statistical metrics (PTI, SIS, CDFR) without machine learning.
SO3 Implement a distributed consensus-based isolation mechanism that requires multi-node agreement before blacklisting, with automatic recovery for sustained good behavior.
SO4 Design and integrate a custom RPL Objective Function (TS-OF) that replaces MRHOF and enforces trust-aware routing by incorporating trust scores into parent selection.
SO5 Validate the framework in Contiki-NG + Cooja simulation, demonstrating >95% accuracy, <2% false positives, and>25% battery life extension.

Methodology

Components at a Glance

Component Technical Name
Component 1 Trust Computation Engine
Component 2 Lightweight IDS - Attack Detection
Component 3 Distributed Isolation & Consensus Manager
Component 4 Trust & Security-Aware RPL Routing (TS-OF)

The research follows an iterative, simulation-driven development methodology structured in four phases:

Phase 1 : Baseline Establishment

A reference RPL network was constructed in Cooja using 11 Contiki-NG motes with standard MRHOF. Baseline metrics (PDR, latency, rank distribution) were recorded to establish a performance reference.

Phase 2 : Component Design and Implementation

Each component was designed independently with clear APIs and implemented in C. This included multi-metric scoring, statistical attack detection windows, node state machines, and the composite metric computation for TS-OF.

Phase 3 : Integration

Components were integrated into a single process on every node, sharing a 10-second timer window. Communication was achieved through shared C variables, UDP multicast for consensus, and Contiki-NG compile-time RPL hooks.

Phase 4 : Validation and Evaluation

Controlled attack scenarios were simulated (Sinkhole, Selective Forwarding, Dual Attacks). Performance was measured via TP/FP counters, Energest power tracking, and structured TSLOG analysis.
